# 实验 02：LoRA SFT v1 — 首轮冷启动（踩坑记录）

> 状态：v1 已完成，识别到三个关键瓶颈，决定以更大 max_length 重新训练｜日期：2026-07-20

## 问题与假设

- **问题：** 用 380 条验收后 SFT 数据对 Qwen3.5-2B 做 LoRA SFT，能否在固定 benchmark 上把严格成功率从 0% 提升到可检测水平？
- **假设：** 只对 assistant token 计算 loss 的 LoRA SFT，能让小模型学会基本的"搜索→打开→选择→购买"工具调用策略。

| 参数 | v1 值 |

| max_length | **12288** |

## 关键踩坑事件

### 坑 1：max_length=24576 OOM
初始设置 24K 直接 OOM（反向传播尝试分配 16 GiB）。降至 12288 才训练成功。

### 坑 2：上下文不匹配导致 SFT 效果被严重低估
训练 max_length=12288，但推理 vLLM max-model-len 最初只设 16384。SFT 模型学会活得更久（平均 19.5 步 vs Base 6.7 步），但 60%+ 轨迹在 16K 上下文时撞 HTTP 400。逐步提升至 32K 才释放模型真实能力。

```
推理上下文  →  error 率
16K         →  67%  （多数成功被扼杀）
24K         →  36%
32K         →  14%
```

### 坑 3：数据利用率低
训练 max_length=12288 只保住了 59%（214/380）的 SFT 数据。后续分析发现：
- p50=10,659 token，p75=16,198，p90=22,410
- 被丢弃的主要是长轨迹（复杂购买任务）—— 恰恰是模型最缺的策略
- 380 条里 224 条≤12K，353 条≤24K（93%），380 条需 50K 才能全保住

| 指标 | Base | SFT v1 | 变化 |
|---|---|---|---|
| strict_success_rate | 0% | ~5% | ↑ 从无到有 |
| invalid_action_limit | 79.5% | 43% | ↓ 减半 |

| error | 14.5% | 14% | ≈ 持平 |
| avg steps | 6.7 | 19.5 | ↑ 3x |

## 失败模式分析

1. **invalid_action_limit (43%)**：模型仍频繁点击不在当前页面的按钮，guard 拦截后连续犯错→终止
2. **max_steps (38%)**：最典型问题——模型学会"不死"但没学会"何时出手"。35 步走满不买，在搜索结果页反复翻页、打开商品但不选规格
3. **error (14%)**：主要是上下文超限 HTTP 400，非策略问题

## 成功的轨迹长什么样

满分轨迹（5%）：平均 **9 步**，流程紧凑：
```
search → open → view_description → prev → back_to_search →
search → open → select_option → buy_now
```
关键特征：第二次搜索更精准、开对商品、选对规格、果断购买。

## 根因分析：为什么 SFT 效果不如预期？

**不是训练配置问题，是数据量和多样性的瓶颈。**

1. **梯度更新仅 81 次** — 对比典型 LoRA SFT（300-5000 次）差一个数量级
2. **214 条数据覆盖的购买策略极其有限** — 强 Teacher 采集的轨迹偏长、偏探索，缺少"果断决策"的范例
3. **eval_loss 平台**：epoch 1→2 降 0.023，epoch 2→3 仅降 0.004 — 模型对现有数据已拟合饱和
4. **41% 数据被丢弃** — 丢失的恰恰是复杂购买逻辑

## 决定：重新训练

三项改进：
1. **max_length：12288 → 24576**（数据利用率 59% → 93%）
2. 验证 24K 是否 OOM（需先做 smoke test）
3. 后续方向：SFT 冷启动足够后上 GRPO 让模型在环境中自己学"何时买"
