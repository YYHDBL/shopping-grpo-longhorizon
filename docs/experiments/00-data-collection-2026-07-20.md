# 实验 00：从 ShopSimulator 收集可训练的单轮购物轨迹

> 状态：已完成当前快照｜日期：2026-07-19 至 2026-07-20｜代码快照：`596e48f` 及其后续数据验收修复

## 这次想回答什么

在不依赖 LLM Judge、Reward Model 或额外 Agent 框架的前提下，能否从真实可交互的 ShopSimulator 中得到一批**可以放心作为 SFT 冷启动**的购物工具调用轨迹？

这里“放心”不等于 Teacher 最后输出了“购买成功”。它必须真的在环境里完成购买，并满足任务的商品类别、属性、规格和价格约束。

## 输入、环境与协议

| 项目 | 本次设置 |
|---|---|
| 商店环境 | ShopSimulator 的结构化 `/api/shop_agent` 接口 |
| 任务来源 | 环境导出的 23,421 个 `task_id`；用户需求在 `reset` 后返回 |
| 任务形式 | 单轮用户需求、多步环境操作；暂不启用 Persona 与多轮澄清 |
| Teacher | OpenAI-compatible 强模型采集；当前发布快照来自 Flash 配置 |
| 工具 | 12 个统一 schema：搜索、打开商品、选规格、查看信息页、翻页、返回、购买、think |
| 轨迹上限 | 最多 35 个环境工具步 |
| 断点续跑 | `raw.jsonl` 是唯一事实来源；相同 task-attempt 不重复采集 |

每条 trajectory 独占一个环境租约。无论购买、模型异常还是工具异常，客户端都会走 `finally` 尝试归还环境；HTTP 失败和环境业务失败分别记录，避免把服务器故障误统计成模型能力差。

## 采集时遇到的真实问题

一开始，最直觉的实现是“模型给一个 tool call，就原样转给环境”。这很快暴露出长轨迹 Agent 的几个小而致命的问题：无参数工具夹带 `asn`、`string` 一类无关字段；模型选完一个规格又返回搜索继续换商品；还有模型把未在当前页面出现的按钮当成可点击目标。

这些问题没有靠人工主观审阅来过滤，而是被放进可复现的规则中：工具参数必须符合 schema，商品号和规格必须来自最近 observation，assistant 每回合只允许一个工具调用。首次选择规格被视为“已确认待买商品”的承诺，后续只允许继续补齐规格或立即购买。这样做牺牲了一些看起来“差一点也能买”的轨迹，却避免把错误的行动模式写入训练数据。

另一个取舍是步数。曾经把上限放得很大，结果失败轨迹会在无效搜索、返回和换商品之间耗尽时间。现在固定为 35 步：它不是最优策略长度的宣言，而是一个防止异常轨迹吞噬预算的工程阈值。超出阈值的轨迹保留在 raw 中供分析，但不进入 SFT。

## 验收器：什么会被留下

轨迹必须同时满足以下条件：

1. 无未处理异常，状态为 `done`；
2. 环境终局同时确认 `done` 和 `over`；
3. 执行过真实 `buy_now`；
4. `reward_detail` 完整，且 `r_type`、`r_att`、`r_option`、`r_price` 全部为 1；
5. 工具调用 JSON、参数、页面动作和单调用顺序均合法；
6. 选规格后没有继续返回搜索、打开新商品或重新搜索。

这是确定性验收：相同 raw 输入永远会得到相同 accepted/rejected 列表和 reject reason。终局 reward、隐藏目标、标准答案和用户画像只用于验收，构造 SFT messages 时不会进入模型上下文；终局 tool observation 会被替换为“购买已完成”。

## 当前结果与数据质量

| 产物 | 数量 | 解读 |
|---|---:|---|
| raw trajectories | 757 | 所有结果，包括失败和异常 |
| accepted | 380 | 严格成功率约 50.2% |
| rejected | 377 | 有结构化拒绝原因，未丢弃证据 |
| SFT JSONL | 380 | 每行是 `messages + tools` |
| 24K 模板可训练行 | 353 | 27 条过长，训练时丢弃而不截断 |

拒绝并不意味着系统没有价值。最多的拒绝信号是没有结束环境/没有购买，其次是规格或属性不满足；这说明后续提高数据量的关键不是盲目扩大 task 数，而是增强 Agent 在商品页上的约束核验能力。完整快照、校验和和拒绝统计见 [输出目录说明](../../outputs/flash_accepted_500_parallel/README.md)。

## 本阶段结论与下一步

**结论：** SFT 冷启动所需的“环境 → Teacher → 严格验收 → 标准 tool-calling JSONL”链路已打通，且数据中不包含用于判分的隐藏字段。380 条 accepted 是一个可复现的起点，不是最终目标；实际在 24K 上下文训练时可用 353 条。

**下一步：** 用未参与 SFT 的固定任务先测 Qwen3.5-2B Instruct 基线，再训练 LoRA SFT。只有拿到相同 benchmark 下的 Base 与 SFT 结果，才讨论是否值得进入 GRPO。
